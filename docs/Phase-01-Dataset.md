# ‚úÖ **Phase 1: Dataset Strategy ‚Äì Detailed Plan**

We need a **realistic yet manageable fraud dataset** that supports:

* **Baseline vs Candidate training** with a **temporal split**
* **Concept drift simulation** (so v1 underperforms on newer data)
* **Fast CPU-only training** (no GPU required)
* **Ground truth labels** for feedback simulation

We‚Äôll achieve this by **enriching the Kaggle Credit Card Fraud dataset** into \~1M rows with **time-based drift injection**.

---

## **1. Dataset Selection**

### **Base Dataset**

* **[Kaggle Credit Card Fraud Dataset](https://www.kaggle.com/mlg-ulb/creditcardfraud)**

  * \~284,807 transactions
  * \~0.17% fraud rate (492 fraud cases)
  * Already has **real statistical structure** (albeit anonymized PCA features)

### **Why Kaggle (Not Fully Synthetic)?**

‚úÖ It‚Äôs **real data** ‚Üí captures realistic fraud/non-fraud feature correlations
‚úÖ We can **extend & perturb** it to simulate drift without generating unrealistic patterns
‚úÖ Easier for others to **reproduce** the demo

---

## **2. Target Scale & Fraud Rate**

We want a **larger dataset (\~1M rows)** to feel more realistic, but not so big that CPU-only training becomes impractical.

* **Final size:** \~1M rows
* **Fraud rate:** **\~1%** (‚âà10k fraud cases) ‚Üí close to reality but still has **enough fraud examples** for training & evaluation

We‚Äôll achieve this by:

* **Oversampling** fraud cases slightly (to increase class signal)
* **Adding Gaussian noise** to non-fraud transactions to expand volume
* **Injecting new fraud patterns for Q1 2024**

---

## **3. Temporal Drift Creation**

To simulate **concept drift**, we‚Äôll partition the dataset into **time windows**:

| Time Window      | Purpose                      | Rows   | Content                             |
| ---------------- | ---------------------------- | ------ | ----------------------------------- |
| **Jan‚ÄìDec 2023** | Baseline v1 training data    | \~700k | ‚ÄúOld‚Äù fraud patterns                |
| **Jan‚ÄìMar 2024** | Candidate v2 additional data | \~200k | Contains **new fraud patterns**     |
| **Feb‚ÄìMar 2024** | Holdout for validation/test  | \~100k | Shows **baseline performance drop** |

---

### **How We Inject Drift in Q1 2024**

1. **New fraudulent merchant categories**

   * Introduce **5 new merchant codes** (e.g., new e-commerce vendors) **seen only in Q1 2024**
   * Baseline v1 hasn‚Äôt seen them ‚Üí more false negatives

2. **Different transaction time-of-day patterns**

   * More fraud between **2am‚Äì5am** in Q1 2024
   * Baseline v1 learned old patterns (e.g., fraud mostly at noon) ‚Üí misses these

3. **Slightly higher fraud transaction amounts**

   * Q1 fraud cases have **higher median amount** ‚Üí baseline underestimates risk

This ensures **baseline recall drops on Feb‚ÄìMar 2024 holdout**, triggering a **need for retraining**.

## **4. Feature Schema**

We‚Äôll simulate a **realistic tabular fraud dataset** with \~30‚Äì50 features.

### **Base Features**

* `txn_amount` ‚Üí transaction amount
* `merchant_category_encoded` ‚Üí encoded merchant type
* `txn_time_of_day` ‚Üí categorical (morning/afternoon/night)
* `geo_risk_score` ‚Üí based on transaction location (0‚Äì1)
* `device_fingerprint_score` ‚Üí device anomaly score (0‚Äì1)
* `account_age_days` ‚Üí age of account in days
* `customer_txn_frequency` ‚Üí average # transactions/day
* `customer_avg_txn_amount` ‚Üí rolling average amount
* `num_chargebacks` ‚Üí historical fraud count for customer

### **Engineered Features**

* `velocity_score` ‚Üí # transactions in last 24h
* `merchant_risk_score` ‚Üí historical fraud rate for merchant
* `geo_distance_from_usual` ‚Üí distance from customer‚Äôs usual transaction location
* `hour_sin`, `hour_cos` ‚Üí cyclical time encoding

### **Drift-Specific Features**

* New merchant categories in Q1
* Nighttime fraud transactions (2‚Äì5am) in Q1
* Higher median fraud amount in Q1

Each transaction will also have:

* `timestamp` ‚Üí for temporal splitting
* `fraud_label` ‚Üí 0 (legit) or 1 (fraud)

## **7. Temporal Splitting Process**

1. **Assign synthetic timestamps** to the enriched dataset

   * Map 700k rows to Jan‚ÄìDec 2023
   * Map 200k rows to Jan‚ÄìMar 2024 (drift period)

2. **Split into train/val/test**:

   ```
   Baseline v1 Train ‚Üí Jan‚ÄìDec 2023 (~700k rows)
   Candidate v2 Train ‚Üí Jan‚ÄìMar 2024 (~900k rows)
   Holdout Test ‚Üí Feb‚ÄìMar 2024 (~100k rows)
   ```

3. Ensure **holdout set contains new fraud patterns unseen by baseline** ‚Üí v1 recall drops, v2 recall improves.


### üîπ Temporal Splitting in Context

* **Baseline v1 Train (Jan‚ÄìDec 2023)** ‚Üí Rows like 1‚Äì3
* **Candidate v2 Train (Jan‚ÄìMar 2024)** ‚Üí Includes rows like 4‚Äì10
* **Holdout Feb‚ÄìMar 2024** ‚Üí Rows 6‚Äì10 used to test drift
